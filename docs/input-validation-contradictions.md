# Input Validation Contradictions

|   # | Fields / values                                                              | Problem                                                                                                                                            | Suggested validation                                                                   | Impact                                                    | Measure                                                                                                       |
| --: | ---------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- | --------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
|   1 | `general.numberOfStories`                                                    | Zero floors produce a negative calculated story height.                                                                                            | Integer `>= 1`                                                                         | Negative geometry and energy values                       | Frontend enforces > 0, validator checks                                                                       |
|   2 | `general.buildingHeight` with omitted `numberOfStories`                      | A small or zero height can infer zero floors through rounding.                                                                                     | Ensure inferred floors are `>= 1`; normally require height `> 0`                       | Negative geometry                                         | Calculation should result in 1 min                                                                            |
|   3 | `general.buildingHeight` + `general.numberOfStories`                         | Both can be supplied while contradicting each other; floor count overrides height in geometry calculations.                                        | Check height against the expected height for the floor count, with an agreed tolerance | Internally inconsistent geometry                          | None                                                                                                          |
|   4 | `general.buildingBaseArea`                                                   | Zero area leads to zero usable floor area and division by zero.                                                                                    | Require `> 0`                                                                          | `Infinity` or `NaN` results                               | Frontend enforces > 0, validator checks                                                                       |
|   5 | Electricity bill base cost + total cost _(frontend)_                         | Base cost can exceed the total bill, resulting in negative derived consumption.                                                                    | `baseCost <= totalCost`                                                                | Negative usage                                            | validator checks, frontend enforces basecost <= total cost                                                    |
|   6 | `heat.userThermalBaseRate` + `heat.userThermalTotalCost`                     | Base rate can exceed total cost.                                                                                                                   | Effective base rate `<= userThermalTotalCost`                                          | Negative thermal consumption                              | see electical                                                                                                 |
|   7 | `heat.userThermalUnitRate` with `userThermalTotalCost`                       | A zero unit rate makes consumption impossible to derive; the bill is currently silently ignored.                                                   | Effective unit rate must be `> 0` when total cost is supplied                          | Supplied bill has no effect                               | frontend enforces and validator checks like suggested                                                         |
|   8 | `electricity.userElectricityConsumption`                                     | Total electricity consumption can be below calculated electrical heating demand.                                                                   | `userElectricityConsumption >= electricalHeatingEnergyDemand`                          | Negative household electricity                            | Expected, no measures.                                                                                        |
|   9 | `outerWall.adjacentWallArea` + `exteriorWallWindows.area` + `outerWall.area` | Adjacent wall and windows can occupy more than the total wall.                                                                                     | `adjacentWallArea + exteriorWallWindows.area <= outerWall.area`                        | Negative free wall area and reduced heat loss             | enforcement dangerous. assume 0 during calculation if area calculates to 0 or below.                          |
|  10 | Omitted `exteriorWallWindows.area` with excessive `adjacentWallArea`         | The automatically estimated window area becomes negative.                                                                                          | `adjacentWallArea <= outerWall.area`                                                   | Negative derived window area                              | like before, enforcement dangerous, assume 0 for wall                                                         |
|  11 | `roofWindows.area` + `roof.area`                                             | Roof-window area can exceed the roof area.                                                                                                         | `roofWindows.area <= roof.area`                                                        | Physically impossible geometry                            | enforcment save in this case, frontend enforces, validator checks.                                            |
|  12 | Roof and roof-window areas generally                                         | Roof heat loss uses the entire roof area and then adds roof-window heat loss.                                                                      | Subtract window area from opaque roof area, assuming `roof.area` is gross area         | Windows are double-counted                                | good catch, fix in calculation.                                                                               |
|  13 | `topFloor.hasAttic = false` + `isAtticHeated = true`                         | An absent attic can be marked as heated.                                                                                                           | `isAtticHeated` may only be true when `hasAttic` is true                               | Contradictory state                                       | no measure, stability of execution takes priority                                                             |
|  14 | `bottomFloor.hasBasement = false` + `isBasementHeated = true`                | An absent basement can be marked as heated.                                                                                                        | `isBasementHeated` may only be true when `hasBasement` is true                         | Different calculations interpret the state inconsistently | treat it same as attic, fix needed in calculator.                                                             |
|  15 | `heat.primaryEnergyCarrier` + gas/storage flags                               | Carrier requirements from configuration are not checked.                                                                                           | Apply the existing `isCarrierCompatible()` check                                       | Impossible carrier/property combinations                  | only change in validator, frontend already enforces                                                           |
|  16 | `heat.primaryEnergyCarrier` + `heat.heatingSystemType`                       | Compatibility is checked only when both fields are explicitly supplied. One explicit field can conflict with the other field's default.            | Resolve defaults first, then validate the resolved pair                                | Unsupported carrier/system combination                    | ignore for now                                                                                                |
|  17 | `bottomFloor.constructionType` + basement state                              | Construction type membership is checked, but `allowedConstructionTypesByHeatedCellar` is not applied.                                              | Validate against the resolved `!hasBasement \|\| isBasementHeated` state               | Semantically invalid construction                         | ignore for now                                                                                                |
|  18 | `preRenovationValues.primaryEnergyCarrier` + `heatingSystemType`             | Both catalogue values can be valid individually but invalid as a pair.                                                                             | Apply `allowedHeatingSystemTypesByCarrier`                                             | Contradictory historical system                           | ignore for now                                                                                                |
|  19 | `preRenovationValues.hadInternalGains` + `heatingSystemType`                 | The submitted flag can disagree with configured system behavior; calculations ignore the submitted flag and derive it again.                       | Require equality, remove the field, or make it authoritative consistently              | Input and calculation disagree                            | ignore, post renovation data is never submitted and is only used for calculation logic for offline scenarios. |
|  20 | `numberOfStories` and year fields                                            | Non-negative frontend number fields still permit decimals.                                                                                         | Require integer floors and integer calendar years                                      | Invalid discrete values                                   | frontend does already enforce integers, validator should check                                                |
|  21 | Component/system year + `general.buildingYear`                               | A renovation or heating-system year can predate the building.                                                                                      | Numeric component/system year `>= buildingYear`                                        | Impossible chronology                                     | frontend should enforce, validator check                                                                      |
|  22 | Any construction/renovation year                                             | Future years can be supplied.                                                                                                                      | Year `<= current year`, if future/planned renovations are not supported                | Impossible chronology                                     | ignore                                                                                                        |
|  23 | `hasInsulation` + `insulationThickness`                                      | `hasInsulation = false` can be combined with a positive thickness; the thickness is ignored. Conversely, true can be combined with zero thickness. | If false, thickness should be absent/zero; if true, require `> 0`                      | Contradictory or ignored input                            | ignore                                                                                                        |

## Recommended Validation Layers

| Validation layer       | Responsibility                                                                                       |
| ---------------------- | ---------------------------------------------------------------------------------------------------- |
| Zod input schemas      | Finite numbers, positive values, integers, basic field constraints                                   |
| `validateInput()`      | Cross-field inequalities, dependent booleans, chronology, resolved default compatibility             |
| Calculation safeguards | Prevent `NaN`, `Infinity`, or negative derived quantities even if validation is bypassed             |
| Frontend               | Immediate user feedback, including electricity bill fields that are not part of the core input model |
